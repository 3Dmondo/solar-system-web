# Bug: Body Labels Overlap UI Controls

Status: Resolved
Date: 2026-05-02

## Summary

Body labels can draw on top of app UI controls, making the HUD, selector rail, playback bar, or other floating controls harder to read or use. Scene labels should remain subordinate to the app chrome even when the labeled body projects near a control surface.

## Environment

- App surface: Solar-system scene labels and floating Phase 4 UI controls
- Browser/device: Unknown
- Route or URL: Main solar-system experience; exact route unknown
- Build/deploy context: After Milestone 11 Phase 4 work; exact branch, commit, and local/deployed context unknown
- Relevant time or data state: Unknown

## Steps To Reproduce

1. Open the solar-system experience with body labels enabled.
2. Navigate, zoom, focus, or wait until one or more projected body labels land near the top-right selector rail, informational HUD, or bottom playback controls.
3. Observe whether the labels render over the UI controls.

## Actual Behavior

Body labels are drawn on top of UI controls. Because labels are clickable, they may also compete with control hit targets when their DOM overlay crosses the same screen area.

## Expected Behavior

Floating app UI should always render above scene labels. Body labels should either appear below the app chrome in the stacking order, be clipped or hidden near reserved UI regions, or otherwise avoid occluding and intercepting UI controls.

## Frequency And Scope

Frequency is unknown. The issue likely depends on camera position, viewport size, active labels, and whether labels project into regions occupied by floating controls. It may be more visible on smaller mobile viewports where controls occupy more screen space.

## Recent Changes

Milestone 11 Phase 4 moved selectors into a top-right control rail and moved simulation playback into a bottom-center bar. Body labels remain HTML overlays created from the Three.js scene via drei `Html`.

## Suspected Area

Likely a UI layering or label-avoidance issue between drei `Html` body labels and the app chrome. `BodyLabel.tsx` renders labels through `Html` and gives the label button `pointerEvents: 'auto'`, while UI controls rely on explicit z-index values such as HUD `10`, playback controls `14`, fullscreen `15`, control rail `16`, and debug overlay `20`. A solver should verify the actual DOM stacking context generated by `Html` and decide whether labels need a lower z-index, disabled pointer events near chrome, or screen-space exclusion zones.

## Open Questions

- Which UI controls were overlapped: top-right rail, open selector panels, HUD, playback bar, or debug overlay? Not isolated to a single control in this fix; the shared label stacking and pointer behavior was corrected for all app chrome.
- Was the overlap observed on mobile, desktop, or both? Unknown from the original report; the fix applies across viewports.
- Does the label visually overlap only, or does it also intercept taps/clicks meant for the UI control? Both risks were addressed: labels now sit below floating chrome and no longer expose clickable label controls.

## Root Cause

drei `Html` labels used the default high z-index range, which can place projected scene labels above the app's absolutely positioned floating chrome. The labels also rendered as clickable buttons with `pointerEvents: 'auto'`, so their DOM overlays could compete with nearby UI hit targets.

## Fix Summary

- Set body-label `Html` overlays to an explicit low z-index range below the floating HUD, selector rail, playback controls, fullscreen button, and debug overlay.
- Changed body labels from clickable buttons to passive text annotations with pointer events disabled.
- Removed the now-unused label selection prop flow from `BodyLabels` and `ExperienceScene`; bodies and indicators remain selectable.
- Added regression coverage for the label layering contract and passive text rendering.
